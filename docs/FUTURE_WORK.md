## Phase 3 vision — auto term-date prediction (user request, June 14)
- [ ] **Auto-fetch academic calendar from school name + program length.** Eventually, given the school + # of years, Marro should look up that school's prior academic calendars, predict future term start/end dates, and pre-fill the year configs — editable by the user, and correctly accounting for years with an extra day (leap years / varying term lengths). The Phase 3 onboarding year-config generator is the seam for this: it currently derives dates arithmetically from a start year, but should be swappable for a calendar source later. **Design principle: do NOT overwhelm the user at setup** — keep first-run minimal, infer/predict as much as possible, let users correct afterward rather than asking up front.
  - **Approach decided (June 14): LLM extraction, lazy + cached — NOT 200 bespoke scrapers.** Writing per-school scrapers (CSS selectors over ~200 LCME+COCA sites in wildly different HTML/PDF/portal formats) is a maintenance treadmill and breaks on every redesign. Instead: (1) when a user from school X onboards with no cached calendar, run a *one-time* fetch for that school — web search → Claude reads the calendar page/PDF and extracts term dates (no selectors); (2) cache the result per-school in Supabase so every future user from X reuses it — coverage grows organically, fetch cost is once-per-school not once-per-user; (3) always editable + framed as an estimate; user corrections feed the per-user predictive tier. Same engine as the aid-letter scan below (Claude vision → structured data). Both need the small backend (server fn / Edge Function) — **build together, Phase 4.**
  - **Provider seam / 3 tiers:** the generator is a swappable date provider — tier 1 heuristic (ship now: weekday-anchored ~early-Aug start, 12-mo leap-aware spans, M3/M4 longer; budgeting only needs the financial-year boundary, not day precision), tier 2 user-corrected (extrapolate from the user's own prior real dates once entered), tier 3 fetched-calendar (seeds tier 2 from the LLM extraction above).
- [x] **Program type (MD / DO) derived from school name, never asked** (June 15) — `degreeForSchool()` regex (all COCA names contain "Osteopathic"). MD-PhD / MD-Master's / DO duals ARE asked (track step) since they can't be inferred; DO duals gated by curated `DO_DUAL` map (expandable; free-text fallback). Stored in `data.program`.
- [ ] **Financial-aid-letter / screenshot upload → auto-fill (user request, June 14).** Let the user upload a financial aid award letter (PDF or screenshot). The system scans it (Claude vision/OCR) and extracts grant/scholarship amounts, tuition, fees, health insurance, living allowance, COA, etc., then auto-fills the year config and creates appropriate spending categories. This is the low-friction alternative to manual entry — pairs with the effortless-expense-capture track. **Phase 4** (Claude advisor) territory; the year-config + categories must stay programmatically fillable so this can drop in.
- **Scalability principle (June 14):** no school gets special-cased. Every school — including Weill Cornell — flows through the same generic generated year-config (financial fields default to 0, filled later via the Aid tab or the aid-letter scan above). Do NOT hardcode any school's tuition/grant numbers as a preset.
