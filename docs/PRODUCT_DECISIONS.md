# Product Decisions

Decision · rationale · date.

- **All CDN `<script>` deps are version-pinned — never use an unpinned/`@latest` URL.** `@babel/standalone` was loaded unpinned; unpkg auto-upgraded it to 8.0.2, whose preset-react defaults to the automatic JSX runtime and injects a top-level `import` into the compiled output. Our app script is a classic `text/babel` script (not a module), so that import is a fatal parse error ("Cannot use import statement outside a module") — React never mounts and the static boot rings hang forever. Hit local + production identically. Pinned Babel to `7.29.7` (classic `React.createElement`, no imports). If migrating to Babel 8 later, must also force `{runtime:"classic"}` on preset-react or switch the script to a real module bundle. (2026-06-20)

- **Neutral dual-theme identity replaces green "candlelight"** — dark = warm near-black `#101210`, light = warm off-white `#F5F4EF`, both with dynamic blobs; marigold + cream + growth rings carry the brand. Rationale: the green ambient layer raised red-green colorblindness concerns and tied the look to one mood; neutral scales to a national audience. (2026-06-11)
- **Blue vs amber is the positive/negative data pair** (`pos #82AEDB` / `neg #E5A23E` dark) — chosen for deuteranopia/protanopia distinguishability; always paired with signs/labels. Info banners moved to slate so blue unambiguously means "positive". (2026-06-11)
- **Destructive stays clay, and is now a separate token from negative data** (`danger` vs `neg`). Amber-as-danger would collide with over-budget semantics; clay buttons are always labeled so the colorblind concern doesn't apply to them. (2026-06-11)
- **Theme persists in the existing `darkMode` sync field**; legacy states were force-migrated to dark once (the old toggle was inert — no user ever chose light); fresh users follow `prefers-color-scheme`. (2026-06-11)
- **De-Cornell is visible-UI only for now** — copy, manifest, icon de-branded; YEAR_CONFIGS amounts/housing notes stay as the user's own data. Full school-agnostic onboarding remains Phase 3. The COA source line stays as data provenance. (2026-06-11) — **SUPERSEDED by the 2026-06-14 "no school is special-cased" decision below.**
- **Icons are hand-drawn ring-derived SVGs**, not an icon library — brand cohesion with the rings logo, no external dependency, marigold dot reserved for savings/live. BRANDS letter-tiles keep their glyphs (content, not chrome). (2026-06-11)
- **Goal progress is circular (RingProgress)**; budget/weekly bars stay linear — rings = growth metaphor for goals, bars = consumption against a cap. (2026-06-11)
- **Fonts are self-hosted variable woff2** (OFL) — an offline PWA must not depend on Google Fonts at runtime. (2026-06-11)
- **Blob health states crossfade stacked gradient layers by opacity** — calm / low-tide (over budget) / 9s marigold bloom (goal funded); gradients themselves are never animated (GPU rule). (2026-06-11)
- **Danger hue is reserved** for destructive/errors; selection is cream (dark) / ink (light). Selection-in-clay made "active" read as "wrong". (2026-06-11)
- **Actual money never rounds** (`fmtA`); plans may round (`fmt`). A money app showing $43 above a $42.50 row erodes trust. (2026-06-11)
- **No native `alert()`/silent submits** — inline `role="alert"` + disabled-until-valid. (2026-06-11)
- **CSV import skips deposits in signed exports**; all-positive files treated as debit-only (covers both bank export styles). (2026-06-11)
- **Logo picker removed** rather than wired up: pre-rebrand artifact, off-brand red WCM marks, external image breaks offline; growth-rings mark IS the brand. (2026-06-11)
- **Renewal flow prefers prefilled continuation** (next cycle date auto-suggested). (2026-06-11)
- **No alarm-red anywhere**; danger is warm clay. Over-budget = "warmth draining" (low-tide), calm by design. (June 2026)
- **Serif money** (Newsreader) is the hero detail — display money ≥ ~16px; small inline amounts stay Inter for legibility. (May 2026, refined 2026-06-11)
- **Destructive confirmations: Cancel dominant**, destructive ghost. Exception: low-stakes reversible removes may use filled clay. (June 2026)
- **App-level actions live in the settings gear menu** (theme, reset, future sync/export) — not as loose header buttons. (2026-06-11)
- **✕ affordances are ghost circles (`XBtn`)** — borderless until hover; boxed ✕s read as form controls. (2026-06-11)
- **Category icons are user-editable at creation AND after** (popover on the icon chip in Categories); picker stays collapsed until asked for. (2026-06-11)
- **Bar-chart hover = dim siblings, no cursor box** — consistent with the donut's hover language. (2026-06-11)
- **Boot screen is a brand moment** (ring bloom + orbiting marigold dot + serif wordmark), pure CSS so it works before React loads on slow connections. (2026-06-11)
- **No card accent bars** — semantic state on cards is carried by the colored figures/pills inside, never by a colored top stripe (it read as a glitch). (2026-06-11)
- **Chart hover detail prefers in-place readouts over floating boxes** — donut center swap instead of tooltip; where tooltips remain, cursor is the themed selection wash. (2026-06-11)
- **Friendly-chart language**: fully rounded bar tops, capped bar width, rounded donut segments — approachable over terminal-finance. (2026-06-11)
- **Custom categories pick a ring-set icon at creation** (`dot` fallback keeps old data valid; icon stored on the category, synced like any field). (2026-06-11)
- **Light mode is toned parchment, not white** — `#ECEAE2`, no brightness boost on glass; "light" must still feel like the same candlelit object. (2026-06-11)
- **Single-file architecture stays** until Phase 2.5b (auth + Supabase); no build step is a feature for this project. (ongoing)

## Phase 2.5b — Auth + Supabase (2026-06-13)
- **Hard login gate** — app requires Google sign-in; no anonymous/local-only mode. One code path, per-user data from day one. (2026-06-13)
- **Keep the blob model** — one jsonb `state` per user in `app_state`, not normalized tables. The 3-way merge engine is transport-agnostic, so Supabase only replaced the Gist transport (`gistFetch/gistWrite` → `stateFetch/stateWrite`); `save`/`checkAndPull`/`resolveConflict`/ConflictModal untouched. Normalizing is a Phase-6 concern, not now. (2026-06-13)
- **localStorage stays as offline cache + merge ancestor; no Realtime** — existing focus/30s polling is enough; Realtime would add a websocket dependency and failure modes for a tiny user base. (2026-06-13)
- **Anon (publishable) key hardcoded in the public repo** — security is RLS, not key secrecy; no build step means env vars aren't an option client-side anyway. Service-role key + Google client secret never in repo. (2026-06-13)
- **School via searchable picker + free-text Other** in a one-time, non-dismissable ProfileModal after first login — clean data for Phase 3/6 benchmarking; embedded curated US MD/DO list (expand to full AAMC/AACOM before public launch). (2026-06-13)
- **Single-file architecture survives 2.5b** — supabase-js loads as a CDN UMD global alongside React/Recharts; still no build step. (2026-06-13)
- **No school is special-cased — every school flows through the same generic generated year-config** (financial fields default to 0, filled later via Aid tab / future aid-letter scan). Supersedes the 2026-06-11 "WCM amounts stay as data" stance: hardcoded `YEAR_CONFIGS`/`DEFAULT_MONTHLY` and the Aid-tab WCM cost-of-attendance table were removed. Rationale: hardcoding any one school's numbers as a preset doesn't scale to a national audience. (2026-06-14)
- **Year dates come from a swappable provider, not a "predictive model" yet** — tier 1 ships now: arithmetic heuristic anchored ~Aug 1, leap-aware, fully editable. Predicting from prior years / scraping calendars needs a data source we don't have at onboarding (no per-school calendar API); that's the tier-2/3 future work. Budgeting only needs the ~12-month financial boundary, not day-precision, so the heuristic is sufficient now. (2026-06-14)
- **Calendar fetch will be LLM extraction (lazy + cached), not 200 bespoke scrapers** — when a user from an uncovered school onboards, fetch+extract that school's calendar once via Claude, cache per-school in Supabase, reuse for all future users. Same engine as the planned aid-letter scan. Both Phase 4 (needs the small backend). (2026-06-14)
- **Progressive setup: onboarding questions are versioned (`SETUP_VERSION` + `SETUP_STEPS`)** — new users answer inline; existing users behind a newly-added question get a focused glass popup for just that step, never a full re-onboarding. v1 grandfathers existing users (they keep their configured years). Rationale: lets us add setup questions over time without disrupting or nagging established users. (2026-06-14)
- **Program type (MD/DO/MD-PhD) is never asked — derive from school name** (LCME vs COCA lists). Deferred until Phase 4 has a consumer to avoid dead code. (2026-06-14)
- **"Key notes" (Customize tab) are derived from the user's own data, not static copy** — built from the active year's grant/tuition/health-ins/housing, the user's Step-exam goals + exam budget, and whether the plan has an extended year. Notes appear only when relevant (e.g. health-insurance note only if `healthIns>0`, extended-year note only if a year has `type==="extended"`) and restate the user's real numbers. Replaced the old hardcoded WCM block (NYC/$5,232 fees/grant-deduction copy). Extended-year Aid-tab banner also de-WCM'd ("costs and grant often differ — check with your aid office") since a generic school may not waive tuition. Rationale: the section should teach the user about *their* plan and stay correct for any school, updating as they fill the app. (2026-06-15)

## Phase 3 — program model (2026-06-15)
- **"Extended year" concept removed entirely** — it was a WCM special-case. Years are now plain numbered years (Year 1, Year 2, …). `generateYearConfigs(startYear, lengthYears)` dropped its 3rd `extendedYear` arg; year objects no longer carry a `type` field. Boot migration converts any legacy `type:"extended"` year into the next numbered year (data preserved, relabeled) and renumbers all years sequentially by start date. (2026-06-15)
- **Dual-degree support (MD-PhD / MD-Master's)** — onboarding step 4 ("Your program") now asks track after school pick: `<degree> only` / `<degree>-PhD` / `<degree> + Master's` / `Other dual degree`. PhD/Master's **field** and **granting institution** (same-as-school toggle or free-text) are optional ("you can add later in Settings"). Program length widened to 3–8 yrs, pre-filled by track (MD 4 / Master's 5 / PhD 8) and adjustable. Stored in the app-state blob as `data.program {degree,dual,phd{field,institution},masters{field,institution}}` — no Supabase schema change. (2026-06-15)
- **Degree (MD/DO) derived from school name, not stored per-entry** — every COCA school name contains "Osteopathic", no LCME school does, so `degreeForSchool(name)` is a regex; satisfies the old "MD/DO from school name" backlog item with zero list maintenance. (2026-06-15)
- **DO dual-degree options gated by a curated `DO_DUAL` map** (per user: DO/PhD & DO duals are rare). MD schools offer PhD+Master's to everyone; DO schools only show the tracks listed in `DO_DUAL` for that school. The map is intentionally incomplete — "Other dual degree" + free-text always remain so onboarding is never blocked; expand as programs are confirmed. (2026-06-15)
- **Program editable post-onboarding in Settings → Program** (`ProgramModal`) — never regenerates years (that's the Aid tab's job), so editing track/fields never wipes budget data. Redo-setup also now captures program. (2026-06-15)
